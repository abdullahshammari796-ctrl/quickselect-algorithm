📖 Read This First — What is Quickselect?

The Problem: Imagine you have a list of 1,000,000 exam scores and you want to know the score of the student ranked exactly 500,000th. The naive approach — sort everything first, then look it up — takes O(n log n) time and does a lot of unnecessary work, because you don't actually need the whole list sorted. You only need one element in its correct position.

Quickselect (invented by Tony Hoare in 1961, the same person who invented Quicksort) solves this order-statistic problem in O(n) average time. It finds the k-th smallest element in an unsorted array without sorting the rest of the array.

The key insight: After a PARTITION step, the pivot element lands at its exact final sorted position. Everything to its left is smaller, everything to its right is larger. If the pivot's position happens to be k, we're done. If not, we know which half contains the answer and only recurse into that one half — discarding the other half entirely. This is what makes Quickselect faster than sorting.

QUICKSELECT(A, low, high, k): if low == high: return A[low] // only one element left → it IS the answer p = PARTITION(A, low, high) // puts pivot at final position p if k == p: // lucky! pivot IS the k-th smallest return A[p] // done — no more recursion needed else if k < p: // answer must be in LEFT half return QUICKSELECT(A, low, p-1, k) // discard right else: // answer must be in RIGHT half return QUICKSELECT(A, p+1, high, k) // discard left PARTITION(A, low, high): // rearranges A in-place pivot = A[high] // pick last element as pivot value i = low - 1 // i = right edge of "≤ pivot" region for j = low to high-1: // j scans every element if A[j] <= pivot: // this element belongs on the left i++; swap(A[i], A[j]) // expand left region, move it in swap(A[i+1], A[high]) // place pivot at its final position return i+1 // pivot's final index — everything left ≤ pivot, right ≥ pivot

public class Quickselect { private static int partition(int[] arr, int low, int high) { int pivot = arr[high]; // last element as pivot int i = low - 1; // right boundary of ≤ pivot region for (int j = low; j < high; j++) { if (arr[j] <= pivot) { i++; int tmp = arr[i]; arr[i] = arr[j]; arr[j] = tmp; // swap } } int tmp = arr[i+1]; arr[i+1] = arr[high]; arr[high] = tmp; return i + 1; } /** Returns k-th smallest element (0-indexed k). */ public static int quickselect(int[] arr, int low, int high, int k) { if (low == high) return arr[low]; // base case int p = partition(arr, low, high); if (k == p) return arr[p]; else if (k < p) return quickselect(arr, low, p - 1, k); else return quickselect(arr, p + 1, high, k); } // ── Example usage ────────────────────────────────── public static void main(String[] args) { int[] arr = {38, 27, 43, 3, 9, 82, 10}; int k = 4; // 4th smallest (1-based) int result = quickselect(arr, 0, arr.length - 1, k - 1); System.out.println("The " + k + "-th smallest: " + result); // → 38 } }

CLO-1 — Asymptotic Analysis
Time Complexity of Quickselect
You can analyze, derive, and justify the time complexity of Quickselect using recurrence relations, best/average/worst case reasoning, and Big-O notation.

📖 How Do We Measure Quickselect's Speed?

What asymptotic analysis means: Instead of counting exact operations (which depends on the hardware, compiler, and input), we describe how an algorithm's runtime grows as the input size n grows. We express this as Big-O notation — the dominant term without constants.

Why Quickselect has three different complexities: The runtime depends entirely on where the pivot lands after each PARTITION call. If the pivot always splits the array perfectly in half, we do less total work. If the pivot is always the worst possible element (always the minimum or maximum), we do the most work. Real inputs fall somewhere in between, which is why we analyze three cases separately.

The geometric series trick: The average-case analysis is the most important. Each time we call PARTITION, it scans the entire active subarray — that costs O(m) where m is the subarray size. Then we recurse into ONE half. So the total work is: n + n/2 + n/4 + n/8 + ... This is a geometric series with ratio 1/2. Its sum is 2n — which is O(n). This is why Quickselect is linear on average, even though it recurses.

Use the interactive charts below to see exactly how these complexity classes grow as n increases. Notice how O(n²) and O(n log n) diverge dramatically from O(n) as n gets large.

Recurrence Relations CLO-1

⚡ Best Case — O(n)

The pivot lands exactly at position k on the first call. We only do one pass of partition (n comparisons) and immediately return.

Recurrence: T(n) = n → O(n)

📊 Average Case — O(n)

With a random pivot, we expect to discard a constant fraction of the array each time. The recurrence resolves to a geometric series:

T(n) = T(n/2) + n

= n + n/2 + n/4 + ... = 2n → O(n)

💀 Worst Case — O(n²)

The pivot is always the minimum or maximum (e.g., already sorted array with last-element pivot). We only remove one element per call:

T(n) = T(n-1) + n

= n + (n-1) + ... + 1 = n(n+1)/2 → O(n²)

Master Theorem Analysis CLO-1

Can we apply the Master Theorem?

The Master Theorem applies to recurrences of the form T(n) = aT(n/b) + f(n).

For average case: T(n) = T(n/2) + O(n) → a=1, b=2, f(n)=n

log_b(a) = log₂(1) = 0. Since f(n) = n = Ω(n^(0+ε)), we are in Case 3 of the Master Theorem.

Result: T(n) = Θ(f(n)) = Θ(n)

For worst case: T(n) = T(n-1) + O(n) — this is NOT in standard Master Theorem form (b must be > 1 as a constant). We solve by expansion (telescoping).

CLO-2 — Algorithm Correctness
Why Does Quickselect Always Find the Right Answer?
Understand the loop invariant, the correctness of PARTITION, and why the recursive selection guarantees the k-th smallest element.

📖 Proving an Algorithm is Correct

What does "correctness" mean? An algorithm is correct if, for every valid input, it produces the right output. With Quickselect, correctness means: given any array A and any valid k, the algorithm always returns the k-th smallest element — never the wrong element, never an infinite loop.

Why proving correctness matters: Correctness is not obvious just because an algorithm "seems to work" on a few examples. We need a rigorous argument that covers all possible inputs — including edge cases like k=1 (minimum), k=n (maximum), arrays with duplicates, or arrays of size 1.

The two-part strategy: We prove PARTITION correct first (using a loop invariant), then prove the full Quickselect correct (using induction). This is the same proof strategy used in CLRS (Introduction to Algorithms) — the standard algorithms textbook.

Loop invariant method: A loop invariant is a property that is (1) true before the loop starts, (2) stays true after each iteration, and (3) implies the correct result when the loop ends. This is similar to how mathematical induction works — you prove a base case and a maintenance step, and the rest follows automatically.

The Key Insight: Partition Invariant CLO-2

What does PARTITION guarantee?

After every call to PARTITION(A, low, high), the pivot element is moved to its exact final sorted position at index p. This is not an approximation — it is guaranteed to be correct. The array is rearranged so that:

  • Every element in A[low..p-1] is ≤ A[p] — these are all smaller than the pivot
  • Every element in A[p+1..high] is ≥ A[p] — these are all larger than the pivot
  • A[p] is in its final sorted position — it will never move again

This means A[p] is exactly the (p+1)-th smallest element in A[low..high]. If p equals our target k, we are done immediately. If not, we know exactly which side contains k and can discard the other side entirely — that is the correctness argument for the recursive step.

Loop Invariant of PARTITION CLO-2

A loop invariant is a property that holds true at the start of every loop iteration. Proving it has three parts: Initialization (true before the first iteration), Maintenance (if true before iteration k, it stays true after), and Termination (when the loop ends, the invariant gives us what we need). This is identical in structure to mathematical induction.

🔒 Invariant (maintained for each j)

At the start of each iteration of the for-loop:

  • All elements in A[low..i] are ≤ pivot
  • All elements in A[i+1..j-1] are > pivot
  • A[high] = pivot (untouched)

✅ Initialization

Before loop starts: i = low-1, j = low. Both subranges are empty — invariant holds trivially.

✅ Maintenance

Each iteration either: keeps A[j] in the right subrange (no swap), or moves A[j] to left subrange (swap with A[i+1], then i++).

✅ Termination

After loop: A[low..i] ≤ pivot, A[i+1..high-1] > pivot. Final swap places pivot at i+1.

Proof of Correctness by Induction CLO-2

Theorem: Quickselect(A, low, high, k) returns A[k] in final sorted order

Base case: low == high. Array has one element — it's both the smallest and largest, so A[low] is the k-th smallest. ✓

Inductive step: Assume Quickselect works on all arrays of size < (high-low+1). After PARTITION returns p:

  • If k == p: A[p] is in its final position, so it IS the k-th smallest. ✓
  • If k < p: The k-th smallest is in A[low..p-1] (smaller subproblem). By induction, Quickselect(A, low, p-1, k) returns it correctly. ✓
  • If k > p: The k-th smallest is in A[p+1..high] (smaller subproblem). By induction, Quickselect(A, p+1, high, k) returns it correctly. ✓

In all cases, the algorithm returns the correct element. ∎

CLO-3 — Algorithm Design Paradigm
Divide & Conquer — But Smarter
Quickselect uses Divide & Conquer, but with a crucial optimization: it only recurses into ONE subproblem, making it fundamentally different from Quicksort.

📖 Algorithm Design Paradigms

What is a design paradigm? A paradigm is a general problem-solving strategy that can be applied to many different problems. The three most important paradigms in algorithms are: Divide & Conquer (break into smaller subproblems, solve recursively), Dynamic Programming (store results of overlapping subproblems), and Greedy (make the locally optimal choice at each step).

Quickselect uses Divide & Conquer — but with a critical twist that makes it more efficient than typical D&C algorithms. Normally, D&C splits a problem into multiple subproblems (like Merge Sort splitting into two halves). Quickselect splits into two halves but only EVER recurses into ONE of them. The other half is permanently discarded because we can prove it cannot contain the answer.

Why not Dynamic Programming? DP requires overlapping subproblems — situations where the same sub-computation appears multiple times. In Quickselect, each recursive call operates on a completely different and disjoint range of the array. No subproblem is ever repeated, so there is nothing to memoize. Using DP here would add space overhead with zero benefit.

Why not Greedy? A greedy algorithm makes the locally best choice without reconsidering it. Quickselect does not make a "best choice" — it makes a deterministic recursive decomposition based on the partition result. The choice of pivot affects performance but not correctness, and we don't select it based on a greedy criterion.

What is Divide & Conquer? CLO-3

📐 Three Steps

1. Divide — Split the problem into subproblems.

2. Conquer — Solve each subproblem recursively.

3. Combine — Merge results back together.

⚡ Quickselect's Twist

Divide — PARTITION splits array around pivot into ≤pivot and ≥pivot.

Conquer — Recurse into only the side containing k. The other side is discarded.

Combine — None needed! We directly return the answer when k==p.

Quickselect vs Quicksort: The Critical Difference CLO-3

How to read the diagram below: The left side shows Quicksort's recursion tree — it fans out into a full binary tree because it must sort both halves. The right side shows Quickselect's tree — it follows a single path downward, because it only recurses into the one half that contains k. The greyed-out nodes on the right are the subtrees that Quickselect never visits. This is the entire reason Quickselect is O(n) instead of O(n log n).

Quicksort — Both Subtrees

After partition at p, Quicksort must recurse on BOTH A[low..p-1] AND A[p+1..high] — it needs to sort everything.

Total work: T(n) = T(k) + T(n-k-1) + n ≈ 2·T(n/2) + n → O(n log n)

Quickselect — One Subtree Only

After partition at p, Quickselect checks which side contains k and recurses into ONLY THAT ONE side — the other side is permanently discarded.

Total work: T(n) = T(n/2) + n → geometric series → O(n)

Greedy vs D&C vs DP — Where does Quickselect fit? CLO-3

🤑 Greedy

Makes the locally optimal choice at each step. Does NOT reconsider. Example: Dijkstra's shortest path selects the closest unvisited node at each step.

Quickselect is NOT greedy — it doesn't make a "locally optimal" choice, it makes a deterministic recursive decomposition.

⚡ Divide & Conquer ✓

Breaks problem into smaller subproblems of the SAME type, solves them recursively. Quickselect fits perfectly: the same problem (find k-th smallest) on a smaller subarray.

📦 Dynamic Programming

Solves overlapping subproblems by storing results in a table. Classic examples: Fibonacci, Longest Common Subsequence, 0-1 Knapsack.

Quickselect is NOT DP — each recursive call operates on a disjoint range, so no subproblem ever repeats. Memoization would waste space with zero benefit.

📖 Choosing the Right Algorithm

Why comparison matters: Knowing one algorithm is not enough. A software engineer must know which algorithm to use in which situation. Quickselect is excellent in many cases, but it has a real weakness: adversarial inputs can push it to O(n²). Understanding when to use it — and when to reach for Median of Medians or Introselect instead — is the difference between a working solution and a production-ready one.

Quickselect vs Quicksort — same partition, different goal: Both algorithms use the exact same PARTITION subroutine. The only difference is what they do after: Quicksort recurses into both halves (goal: sort everything), while Quickselect recurses into one half (goal: find one element). This single difference cuts the complexity from O(n log n) to O(n) on average.

The Median of Medians algorithm (also called BFPRT, after its five inventors) solves the worst-case problem by choosing a pivot that is guaranteed to fall within the middle 30%–70% of the array. This ensures each recursive call eliminates at least 30% of elements, giving a worst-case O(n) guarantee. The tradeoff: the constant factor is much larger (~5× slower than Quickselect in practice), so MoM is rarely used unless worst-case guarantees are required.

Introselect (used in C++ std::nth_element) is the pragmatic hybrid: run Quickselect normally, but track recursion depth. If depth exceeds O(log n), switch to Median of Medians. This gives Quickselect's speed in the common case with MoM's worst-case guarantee — the best of both worlds.

Algorithm Selection Guide CLO-4

The right choice depends on your constraints. Ask yourself: Do I need a worst-case guarantee? Do I need to sort all the data anyway? Is n large enough for asymptotic differences to matter? Use the cards below as a decision framework.

🥇 Use Quickselect when...

  • You need ONE specific order statistic
  • Average-case performance is acceptable
  • In-place modification is OK
  • Real-world data (random-ish inputs)

🥈 Use Median of Medians when...

  • Worst-case guarantee is required
  • Adversarial inputs possible
  • O(n) worst-case is a hard requirement
  • (Constant factor is larger than Quickselect)

🥉 Use Sort+Index when...

  • You need MANY order statistics
  • The sorted order is needed anyway
  • Simplicity is paramount
  • n is small (constant factors dominate)

Quickselect vs Quicksort — Structural Comparison CLO-4

These two algorithms share identical PARTITION code. Read both side by side and find the one line that is different. Quicksort has two recursive calls at the bottom. Quickselect has two recursive calls but only one ever executes — the other branch is taken instead. That single structural difference is responsible for the entire O(n log n) vs O(n) complexity gap.

Quicksort — must sort everything

QUICKSORT(A, low, high): if low < high: p = PARTITION(A, low, high) QUICKSORT(A, low, p-1) // ← recurse left QUICKSORT(A, p+1, high) // ← AND recurse right // Both halves! → O(n log n)

Quickselect — only needs one answer

QUICKSELECT(A, low, high, k): if low == high: return A[low] p = PARTITION(A, low, high) if k == p: return A[p] elif k < p: return QS(A, low, p-1, k) // ← OR left else: return QS(A, p+1, high, k) // ← OR right // ONE half only! → O(n) avg
